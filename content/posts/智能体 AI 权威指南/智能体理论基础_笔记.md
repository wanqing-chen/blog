# 智能体理论基础笔记

## 1. 智能体的第一性原理：理性与效用

### 理性智能体 (Rational Agent) 定义

Stuart Russell 和 Peter Norvig 在《人工智能：一种现代方法》(AIMA, 1995) 中定义：

> **理性智能体**：对于每一个可能的感知序列，根据已知的感知序列和内置的知识，选择能使其**预期效用 (Expected Utility)** 最大化的行动。

### 核心决策公式

$$a^* = \arg\max_{a \in A} \mathbb{E}[U(s') | s, a]$$

| 符号 | 含义 |
|------|------|
| $a^*$ | 最优行动 |
| $s$ | 当前状态 |
| $a$ | 候选行动 |
| $s'$ | 执行行动后可能到达的新状态 |
| $U(s')$ | 新状态的效用值 (Utility) |

### 从强化学习到大模型智能体

| 传统强化学习 | LLM 智能体 |
|------------|-----------|
| 显式定义奖励函数 (Reward Function) | **隐式效用**：RLHF 训练已内化符合人类价值观的效用函数 |
| 手动计算效用 | **指令即目标**：系统提示词定义当前任务的效用边界 |

> **设计启示**：设计智能体的提示词时，本质上是在定义 $U(s)$。如果提示词模糊不清，智能体就无法计算出哪个行动能带来最大效用，从而导致行为混乱。

## 2. 核心循环：从 MDP 到 POMDP

### 马尔可夫决策过程 (MDP)

智能体与其环境的交互在数学上通常被建模为 **马尔可夫决策过程 (MDP)**。

### 部分可观测马尔可夫决策过程 (POMDP)

在真实世界应用中，由于智能体的上下文窗口与感知能力存在物理边界，它更准确地是一个 **POMDP**。

#### MDP vs POMDP

| 类型 | 视角 | 说明 |
|------|------|------|
| **MDP** | 上帝视角 | 智能体能看到世界的**全部**状态（如：下围棋，棋盘完全透明） |
| **POMDP** | 真实世界 | 智能体只能看到世界的**一部分**（如：编程智能体只能读取打开的文件，看不到运行时内存） |

#### 限定上下文边界

优秀的智能体工程，本质上是用约束条件在 POMDP 的混沌中切分出一个"确定的、边界清晰的小型 MDP"：

- 限制每次只处理当前函数或文件（而非整个仓库的语法树）
- 通过 RAG 对客观事实进行投影，降低环境状态的不确定性

## 3. "记忆"的理论本质：状态估算

### 信念状态 (Belief State)

在 POMDP 中，仅凭当前观测 $o_t$ 是无法决策的。智能体根据观测构建对世界的猜测，即 **信念状态**：

$$b_t = f(b_{t-1}, a_{t-1}, o_t)$$

| 符号 | 含义 |
|------|------|
| $b_t$ | 当前的信念状态 |
| $b_{t-1}$ | 上一刻的信念状态 |
| $a_{t-1}$ | 上一刻执行的行动 |
| $o_t$ | 最新的观测 |

### 具体示例

假设一个编程智能体正在分析项目依赖：

| 步骤 | 内容 |
|------|------|
| $b_0$（初始信念） | "我不知道这个项目用什么框架" |
| $a_0$（行动） | 执行 `read_file("requirements.txt")` |
| $o_1$（观测） | 文件内容显示 `flask==2.0.1, sqlalchemy>=1.4` |
| $b_1$（更新后的信念） | "这是一个使用 Flask 和 SQLAlchemy 的 Python Web 项目" |

### 核心结论

- **上下文 = 信念状态的文本化表示**
  - 当智能体把"运行报错信息"加入上下文时，它就在更新信念状态
- **记忆即状态**：RAG 和长上下文窗口的本质，是帮助智能体更好地估计信念状态、补足关键上下文，从而降低不确定性

## 4. 决策架构：从反应式到认知式

### 反应式架构：Reflex Agent（系统 1）

| 项目 | 说明 |
|------|------|
| **原理** | 感知 → 行动 |
| **公式** | $Action = Policy(State)$ |
| **表现** | 你问一句，LLM 答一句。没有内部独白，没有思考过程。 |
| **局限** | 无法解决需要多步推理的复杂问题。 |

### 认知式/审慎架构：Deliberative Agent（系统 2）

| 项目 | 说明 |
|------|------|
| **原理** | 感知 → **构建模型 → 规划/推理** → 行动 |
| **关键** | 在输出行动之前，先进行内部的"思考时间" |

### 思维链 (CoT) 的理论意义

CoT 不仅仅是提示词技巧，它在理论上通过生成中间 Token，增加了模型用于计算当前问题的 **测试时计算量 (Test-Time Compute)**：

- 将 $P(y|x)$ 拆解为 $P(z|x) \cdot P(y|x, z)$，其中 $z$ 是中间推理步骤
- 通过这种方式，非结构化的推理过程被**显式化、结构化**

## 5. 环境建模：PEAS 框架

| 维度 | 定义 | 现代智能体设计考量 |
|------|------|------------------|
| **P**erformance | **性能度量** | 目标函数。需转化为**量化评估指标**（代码通过率、API 消耗成本）和 CI 卡点阈值 |
| **E**nvironment | **环境** | 决定架构复杂度：<br>• 完全/部分可观测：是否需要强记忆模块？<br>• 确定/随机：工具调用失败是否需要重试？<br>• 静态/动态：思考时环境会变吗？ |
| **A**ctuators | **执行器** | 智能体的"手" = **Tools / Function Calling**。所有输出（文本、JSON、API 请求）都是行动 |
| **S**ensors | **传感器** | 智能体的"眼"。除用户文字外，还可以是图像 (Vision)、LSP 报错信息、网页 DOM 树等 |

## 6. 理论到设计的映射

| 经典 AI 理论概念 | 现代 LLM 智能体中的近似对应 | 说明 |
|----------------|--------------------------|------|
| **策略函数 Policy $\pi(a\|s)$** | **LLM + 外层控制逻辑** | 给定上下文 $s$，模型生成候选行动；最终能否执行受 harness、权限与工具路由约束 |
| **信念状态 Belief State $b_t$** | **Context Window / Memory** | 维护当前对话历史、变量状态、环境观测的文本快照 |
| **规划 Planning / Search** | **CoT / ReAct / Planner 模块** | 通过中间推理步骤、任务拆解或外部 planner 模块模拟搜索和规划过程 |
| **感知 Perception $o_t$** | **输入通道与观测编码** | 将多模态信号（视觉、代码、日志）编码为 Token 序列或结构化上下文 |
| **行动 Action $a_t$** | **Function Calling / Tool Use** | 输出结构化请求来触发外部代码执行 |
| **环境反馈 / 观察更新** | **Tool Outputs / Runtime Feedback** | 工具执行结果、编译器报错、人工反馈等会更新下一轮上下文（不天然等同于强化学习中的标量 reward） |

## 核心概念关系图

```
真实世界 (State S)
      ↓ 观测 O
感知 (Sensor) → 信念状态 (Belief b) → 策略 (Policy π) → 执行器 (Actuator) → 改变环境
                        ↑
                        └── 更新 (基于行动 a 和新观测 o)
```

## 要点总结

1. **理性智能体**：目标是最大化预期效用，设计提示词本质上是定义效用函数
2. **POMDP**：智能体在部分可观测环境中工作，需要通过"记忆"维护信念状态
3. **信念状态**：上下文就是信念状态的文本化表示，RAG/长上下文是信念估计的工程实现
4. **系统 1 vs 系统 2**：反应式（Reflex）vs 认知式（Deliberative），后者引入规划/推理
5. **PEAS 框架**：分析智能体环境时从 Performance、Environment、Actuators、Sensors 四个维度入手
