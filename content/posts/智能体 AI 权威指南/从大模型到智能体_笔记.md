# 从大模型到智能体：核心演进笔记

## 1. 从生成式 AI 到智能体 AI

### 核心转变

- **传统 LLM**：博学的思想家，停留在"输入-输出"范式。能看能说，但未必能稳定地"做"。
- **智能体 AI**：赋予 LLM "手"（工具）、"眼"（感知）、"记忆"，将思想家变为实干家。

### 本质区别

| 维度 | 传统 LLM | 智能体 AI |
|------|---------|----------|
| **目标** | 概率预测（生成文本） | 目标导向（达成任务） |
| **状态** | 无状态（每次请求独立） | 有状态（持久化记忆） |
| **执行** | 单次推理 | 主动循环（思考→行动→观察→...） |

## 2. 核心差异详解

### 2.1 目标导向 vs 概率预测

- **LLM**：训练目标是预测下一个Token的概率分布，问答只是训练数据的统计相关性
- **智能体**：内置核心指令"你是一个能够解决问题的助手"，先思考再行动

### 2.2 有状态 vs 无状态

智能体维护的持久化状态包括：
- **长期记忆**：记得用户的历史偏好和上下文
- **任务状态**：追踪任务进度（如"订机票：已搜索，等待确认"）
- **环境状态**：知道当前目录下有哪些文件

### 2.3 主动循环 vs 单次推理

- **LLM**：输入 A → 输出 B（线性，一次性）
- **智能体**：目标 → 感知与规划 → 行动/工具调用 → 外部环境 → 观察结果 → 循环直到完成

## 3. 认知升级：系统 1 与系统 2

### 映射关系

| 人类思维 | AI 映射 | 特征 |
|---------|--------|------|
| **系统 1**（直觉、快速、无意识） | LLM 直接生成 | 快而创造，但易产生幻觉 |
| **系统 2**（逻辑、慢速、有意识） | 智能体规划与反思 | 拆解任务、执行、检查、回退重试 |

### 关键说明

> 边界正在模糊：OpenAI o 系列等推理增强模型开始在内部具备部分"系统 2"能力，但从工程落地视角看，长周期任务的持久化记忆、异步工具调用管理仍属于智能体工程框架（外在"系统 2"）不可替代的职能。

### 智能体架构本质

在 LLM（强大的"系统 1"）外层，用工程手段（上下文工程、记忆系统、工具系统）包裹了一层"系统 2"的逻辑外壳。

## 4. 为什么现在爆发？

三大要素的"涌现"：

1. **推理能力的临界点**：上下文学习与多步推演变得更可靠，使"让模型自己写计划"成为可能
2. **标准化的交互协议**：Function Calling / Tool Use → MCP（模型-工具/数据源连接标准化）→ A2A（agent间互操作）
3. **记忆与上下文的扩展**：RAG 与更长上下文窗口使智能体能处理海量信息

## 5. 早期智能体的教训

| 问题 | 描述 |
|------|------|
| **幻觉传递** | 模型在工具参数中产生的幻觉会被实际执行，导致不可预期的副作用 |
| **无限循环** | 缺乏最大步数限制，陷入"重试-失败-重试"的死循环，消耗大量 Token |
| **上下文爆炸** | 每次工具调用结果追加到上下文，Token 成本呈线性增长 |
| **错误处理缺失** | 工具返回错误时，系统不知道如何优雅降级 |

## 6. 关键里程碑

### 早期阶段：Chatbot 元年
- 对话式大模型广泛普及，工具使用进入大众视野
- AutoGPT 等开源项目展示 LLM 自主循环的可能性

### 架构成型与推理提升
- 推理增强模型出现，多步推理更可靠
- 计算机使用能力（Computer Use）：模型能在受控环境下操作 GUI
- 统一连接协议出现（MCP）

### 标准化与生产化
- 工具、资源、提示模板接口规范化
- 治理层独立化：权限、审计、人工审批沉到 host、sandbox 与策略层

### 生态互操作与扩展
- 长上下文与计算机使用能力增强
- 代码专用模型推动"从需求到可运行系统"的自动化
- MCP、A2A 等协议成熟，生态更易迁移与复用

## 7. 核心演进总结

### 维度一：连接方式

- **早期**：每个工具手动编写调用代码（烟囱式）
- **最新**：标准化 + 动态发现，运行时发现"你能提供什么？格式？权限？"

### 维度二：决策逻辑

- **早期**：Router（分发器），多步骤任务易"断片"
- **最新**：认知架构
  - 慢思考（System 2）：思维链（CoT）规划
  - 分层规划：Goal → Strategy → 原子技能
  - 自我进化：反思内存记录失败教训

### 维度三：安全控制

- **早期**：System Prompt 叮嘱（易被指令注入攻破）
- **最新**：独立策略引擎 + 权限系统
  - 硬拦截：安全规则在 Sandbox 而非模型内
  - 零信任架构：每次技能调用实时校验上下文

### 维度四：协作模式

- **早期**：单体多技能（Context Window 被技能描述塞满）
- **最新**：多智能体协作（MAS）+ 编排工作流
  - 社会化分工：经理智能体、文案智能体、代码审核智能体
  - 图式工作流定义智能体互动

## 核心演进总结表

| 维度 | 痛点 | 解决方案 | 关键技术 |
|------|------|---------|---------|
| **集成** | 烟囱式开发，扩展难 | 工业级协议，即插即用 | Tool-aware |
| **智商** | 走一步看一步，易出错 | 三思而后行，分层思考 | Reasoning LLM, Reflection |
| **安全** | 容易被"忽悠"，防不住 | 权限隔离，硬性审计 | Policy as Code, Zero Trust |
| **效率** | 臃肿、思维混乱 | 小而美，协同作战 | MAS, State-sharing |

## 附：行业前沿观点

**编程形态的演进**（TiDB CTO 黄东旭，QCon 2026）：

| 时代 | 思考载体 | 执行载体 |
|------|---------|---------|
| 最初 | 机器码 | 机器码（合一） |
| 编程语言时代 | 代码 | 硬件（分离） |
| Agent 时代 | Goal + Context + Constraints | 代码（与硬件同级） |

**核心转变**：人的工作从"编写代码逻辑"转向"定义目标、提供上下文、设定约束"。
