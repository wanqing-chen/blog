# 核心组件笔记：大脑、感知、行动与记忆

## 1. 智能体核心架构概览

智能体核心架构由四个部分组成：

| 组件 | 功能 | 说明 |
|------|------|------|
| **大脑 (Brain)** | 规划与决策 | 负责拆解、规划、反思 |
| **感知 (Perception)** | 信息输入与处理 | 文本、多模态、环境状态 |
| **行动 (Action/Tools)** | 与世界交互 | 工具与输出接口 |
| **记忆 (Memory)** | 经验存储 | 短期与长期记忆 |

### 智能体核心组件架构图

```
用户指令
    ↓
感知 → 大脑 ←→ 记忆
    ↓                      ↓
行动 → 策略网关与沙箱 → 目标系统
       (TCB 拦截层)
```

> **可信计算基 (TCB)**：生产级智能体的必备措施。在将行动下发至目标系统前，必须经过包含硬编码策略引擎、权限校验和沙箱隔离的 TCB 拦截层。

---

## 2. 思考三维框架

任何 Agent 相关的工作都可以归入以下三个维度：

| 维度 | 说明 | 包含内容 |
|------|------|---------|
| **Goal（目标工程）** | 如何定义和表达 Agent 应当掌握的工作流和领域知识 | Skill、人机交互 |
| **Context（上下文工程）** | Agent 执行时看到什么 | Harness Engineering、Memory、RAG、信息环境 |
| **Constraints（约束）** | Agent 的执行边界 | Sandbox、CICD/Lint、Agent Infra |

> **核心洞察**：Agent 的能力不只取决于模型的智力，更取决于这三个维度的设计完整性和平衡。约束层的成熟度往往成为进入生产环节的瓶颈。

---

## 3. 大脑：核心控制器

大脑由 **LLM** 扮演，是整个系统的 CPU，所有逻辑推理、指令遵循和决策都在这里发生。

### 静态维度：角色设定

| 方法 | 说明 |
|------|------|
| **身份锚定** | "你是一个拥有10年经验的资深Python架构师，性格严谨"——影响说话语气，甚至提高代码质量 |
| **少样本示例 (Few-shot)** | 提供3-5个完美互动样本，通过**上下文学习 (In-context Learning)** 让大脑迅速领悟任务模式 |

### 动态维度：推理能力

| 能力 | 说明 |
|------|------|
| **任务拆解 (Decomposition)** | 将用户模糊指令拆解为原子任务列表 |
| **规划 (Planning)** | 预判步骤的依赖关系 |
| **反思 (Reflection)** | 高级功能，在输出结果前进行自我审视 |

### 选型提示

| 模型类型 | 适用场景 |
|---------|---------|
| **高能力模型** | 充当"主脑"，处理复杂逻辑、规划与反思 |
| **轻量模型** | 充当"小脑"或"脊髓"，处理分类、摘要、路由等轻任务，成本更低 |

## 4. 感知：智能体的感官

感知模块负责将环境中的 **非结构化信息** 转化为大脑可以理解的 **结构化信号**（文本或嵌入）。

### 感知类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **基础感知** | 文本输入 | 用户指令、API 返回的 JSON 数据、错误日志 |
| **高级感知** | 多模态 | 视觉 (Vision)、听觉 (Audio) |
| **深度感知** | 环境状态 | IDE 打开的文件路径、git diff、LSP 语法错误 |

### 多模态感知详解

- **视觉 (Vision)**：利用多模态模型，"看"懂网页内容与界面元素，从截图中定位按钮与表单
- **听觉 (Audio)**：通过语音识别模型，接入电话或会议系统，实时转录语音并回应

### 编程智能体的深度感知

- 读取 IDE 当前打开的文件路径
- 通过 `git diff` 感知代码变更
- 通过 LSP (Language Server Protocol) 感知代码中的语法错误红线

---

## 5. 行动：手与工具

**行动空间 (Action Space)** 定义了智能体的能力边界。

### 核心观点

> 没有工具的智能体只是一个只会聊天的缸中之脑。

智能体通过工具与外部系统交互：
- 信息检索
- 代码执行
- 文件操作
- 系统命令

这些工具让模型能够：
- 突破知识截止
- 获取实时数据
- 执行可靠的计算
- 对真实世界产生影响

---

## 6. 记忆：经验的载体

LLM 本身是无状态的，**上下文窗口 (Context Window)** 昂贵且有限。记忆模块旨在解决"健忘"问题。

### 记忆类型与对应关系

| 记忆类型 | 技术实现 | 特征 |
|---------|---------|------|
| **感觉记忆** | 原始输入流 | 所有用户提示词、系统消息最初存储地 |
| **短期记忆** | LLM 上下文窗口 | 存放当前对话历史，像 RAM，断电即失 |
| **长期记忆** | 向量数据库 (Vector DB) / 外部存储 | 像硬盘，可持久化 |

### 短期记忆的限制

- 即便模型支持更长上下文窗口，塞入太多无关信息仍会降低推理精度
- **中间丢失 (Lost in the Middle)** 现象：上下文中间的信息容易被忽略

### 长期记忆与 RAG

- **检索增强生成 (RAG)**：当记忆太多放不下上下文时，智能体会先在向量库中检索与当前问题最相关的片段，只把这几条"回忆"加载到短期记忆
- **经验积累**：智能体可以将成功案例总结成经验存入长期记忆，下次遇到类似问题时表现更好

## 要点总结

| 组件 | 核心职责 | 关键技术 |
|------|---------|---------|
| **大脑** | 规划与决策 | LLM、角色设定、CoT、Few-shot、反思 |
| **感知** | 信息输入 | 文本、多模态 (Vision/Audio)、环境状态 |
| **行动** | 世界交互 | Tools/Function Calling |
| **记忆** | 经验存储 | 短期=Context Window，长期=Vector DB/RAG |

| 三维框架 | 说明 |
|---------|------|
| **Goal** | 目标工程：Skill + 人机交互 |
| **Context** | 上下文工程：Harness + Memory + RAG |
| **Constraints** | 约束：Sandbox + CICD/Lint + Agent Infra |
